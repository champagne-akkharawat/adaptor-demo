# Unified Message Schema ŌĆö Proposals

Five schema proposals based on best practices in event-driven architecture, API design, and enterprise messaging systems. Platform-specific details deliberately excluded at this stage.

---

## Proposal 1: CloudEvents Envelope Pattern

**Inspired by**: [CloudEvents spec](https://cloudevents.io/), AWS EventBridge schema registry.

**Philosophy**: Strict separation between *routing/meta* and *content*. The envelope is platform-agnostic by definition; only `data` varies. Optimised for event queues (SQS, SNS, EventBridge).

```json
{
  "specversion": "1.0",
  "id": "01HZ9KXYZ...",
  "source": "adaptor/line",
  "type": "message.inbound.text",
  "time": "2026-03-16T10:00:00Z",
  "datacontenttype": "application/json",

  "data": {
    "sender": {
      "id": "u_abc123",
      "display_name": "Jane Doe",
      "avatar_url": "https://..."
    },
    "conversation": {
      "id": "c_xyz789",
      "type": "direct"
    },
    "content": {
      "type": "text",
      "text": "Hello world"
    },
    "attachments": [],
    "reply_to": null
  }
}
```

**Envelope fields:**

| Field | Type | Description |
|---|---|---|
| `specversion` | string | Always `"1.0"` ŌĆö CloudEvents version |
| `id` | ULID/UUID | Globally unique event ID, generated by adaptor |
| `source` | string | `adaptor/{platform}` ŌĆö which adaptor emitted this |
| `type` | string | Dot-separated event type: `message.inbound.{content_type}` |
| `time` | RFC3339 | When the event was emitted by the adaptor |
| `datacontenttype` | string | Always `application/json` for this system |

**`data` fields:**

| Field | Type | Description |
|---|---|---|
| `sender.id` | string | Platform-scoped user ID |
| `sender.display_name` | string? | Human-readable name if available |
| `sender.avatar_url` | string? | Profile picture URL if available |
| `conversation.id` | string | Channel/thread/DM identifier |
| `conversation.type` | enum | `direct`, `group`, `channel` |
| `content.type` | enum | `text`, `image`, `video`, `audio`, `file`, `location`, `sticker`, `unsupported` |
| `content.text` | string? | Plain text body (present when type is `text`) |
| `content.url` | string? | Media URL (present for media types) |
| `content.mime_type` | string? | MIME type for media |
| `content.caption` | string? | Caption for media |
| `attachments` | array | Secondary content items using same shape as `content` |
| `reply_to` | string? | `id` of the message being replied to |

**Pros:**
- Interoperable ŌĆö any CloudEvents-compliant consumer can read the envelope
- Clean contract between adaptors and consumers
- `type` field enables consumer-side filtering at queue level without deserialising `data`

**Cons:**
- `data` is still ad-hoc JSON ŌĆö no schema enforcement without a separate registry
- `specversion` is noise for internal-only systems

## Proposal 2: Typed Content Variant (Discriminated Union / Sum Type)

**Inspired by**: Protocol Buffers `oneof`, GraphQL union types, Rust enum patterns.

**Philosophy**: The message content is a *discriminated union* ŌĆö exactly one content variant is present and it is fully typed. Avoids nullable fields; each content type has its own dedicated shape. Suited for strongly-typed codegen (Go structs, TypeScript discriminated unions, Protobuf).

```json
{
  "id": "01HZ9KXYZ...",
  "platform": "line",
  "platform_message_id": "abc123",
  "direction": "inbound",
  "timestamp": "2026-03-16T10:00:00Z",

  "sender": {
    "id": "Uabc123",
    "name": "Jane Doe"
  },

  "conversation": {
    "id": "Rxyz789",
    "type": "direct"
  },

  "content": {
    "kind": "text",
    "text": {
      "body": "Hello world",
      "preview": null
    }
  },

  "reply_to": null,
  "idempotency_key": "line:abc123"
}
```

**Content variant shapes** ŌĆö exactly one `kind` is present, and only its corresponding field is populated:

```json
{ "kind": "text",     "text":     { "body": "...", "preview": null } }
{ "kind": "image",    "image":    { "url": "...", "mime_type": "image/jpeg", "width": 1280, "height": 720, "caption": null } }
{ "kind": "video",    "video":    { "url": "...", "mime_type": "video/mp4", "duration_secs": 30, "thumbnail_url": "..." } }
{ "kind": "audio",    "audio":    { "url": "...", "mime_type": "audio/ogg", "duration_secs": 12 } }
{ "kind": "file",     "file":     { "url": "...", "mime_type": "...", "filename": "report.pdf", "size_bytes": 204800 } }
{ "kind": "location", "location": { "lat": 35.6762, "lng": 139.6503, "label": "Tokyo Station" } }
{ "kind": "sticker",  "sticker":  { "sticker_id": "...", "package_id": "...", "url": null } }
{ "kind": "template", "template": { "template_id": "order_confirm", "variables": { "order_id": "ORD-99" } } }
{ "kind": "unsupported", "unsupported": { "raw_type": "custom_widget", "description": "Platform-specific content not mapped to canonical schema" } }
```

**Top-level fields:**

| Field | Type | Description |
|---|---|---|
| `id` | ULID | Canonical ID generated by adaptor |
| `platform` | enum | Source platform identifier |
| `platform_message_id` | string | Platform's own message ID |
| `direction` | enum | `inbound` \| `outbound` |
| `timestamp` | RFC3339 | Platform-reported send time |
| `sender.id` | string | Platform-scoped user ID |
| `sender.name` | string? | Display name if available |
| `conversation.id` | string | Channel/DM identifier |
| `conversation.type` | enum | `direct`, `group`, `channel` |
| `content.kind` | enum | Discriminator ŌĆö determines which sibling field is present |
| `reply_to` | string? | Canonical `id` of parent message |
| `idempotency_key` | string | `{platform}:{platform_message_id}` ŌĆö used for dedup at DB/queue level |

**Pros:**
- No nullable fields within a content variant ŌĆö every present field is meaningful
- Trivially maps to Go `interface{}` + type switch, TS discriminated union, or Protobuf `oneof`
- `idempotency_key` is explicit and composite ŌĆö prevents double-processing across restarts
- Minimal top-level footprint ŌĆö fast to serialise onto SQS

**Cons:**
- Adding a new content type requires schema changes in all consumers (coordinated deploy)
- `unsupported` is a catch-all that can hide mapping gaps if not monitored

## Proposal 3: Event Sourcing / Append-only Log Model

**Inspired by**: CQRS/Event Sourcing (Greg Young), Kafka message conventions, EventStoreDB schema patterns.

**Philosophy**: A message is an *immutable domain event* that records a fact ŌĆö "a message was received". It carries full causation and correlation context, a stream partition key, and a sequence position. The schema is designed for an append-only log (Kafka, SQS FIFO, Kinesis) where order, dedup, and replay are first-class concerns. The canonical message is never mutated; corrections are new events.

```json
{
  "event_id": "01HZ9KXYZ...",
  "event_type": "MessageReceived",
  "event_version": 1,
  "occurred_at": "2026-03-16T10:00:00Z",
  "recorded_at": "2026-03-16T10:00:01Z",

  "stream": {
    "id": "conversation:line:Rxyz789",
    "sequence": 142
  },

  "causation_id": null,
  "correlation_id": "sess_01HZ...",

  "payload": {
    "platform": "line",
    "platform_message_id": "msg123",
    "direction": "inbound",

    "sender": {
      "id": "Uabc123",
      "name": "Jane Doe"
    },

    "content_type": "text",
    "content": {
      "text": "Hello world"
    },

    "reply_to_event_id": null
  },

  "metadata": {
    "adaptor": "line-adaptor",
    "adaptor_version": "1.0.0",
    "schema_version": "messages/v1",
    "idempotency_key": "line:msg123"
  }
}
```

**Top-level event fields:**

| Field | Type | Description |
|---|---|---|
| `event_id` | ULID | Globally unique event ID ŌĆö monotonic, sortable |
| `event_type` | string | PascalCase domain event name: `MessageReceived`, `MessageUpdated`, `MessageDeleted` |
| `event_version` | int | Schema version of this event type ŌĆö enables consumer-side migration |
| `occurred_at` | RFC3339 | When the event *happened* (platform time) |
| `recorded_at` | RFC3339 | When the adaptor *recorded* it ŌĆö delta = processing latency |
| `stream.id` | string | Partition key: `{entity_type}:{platform}:{id}` ŌĆö routes to the same consumer partition |
| `stream.sequence` | int? | Position within the stream ŌĆö set by the queue/broker, null if not yet assigned |
| `causation_id` | ULID? | `event_id` of the event that *caused* this one (e.g. a command that triggered a reply) |
| `correlation_id` | string? | Groups related events across a user session or workflow |

**`payload` fields:**

| Field | Type | Description |
|---|---|---|
| `platform` | enum | Source platform |
| `platform_message_id` | string | Platform's ID ŌĆö used with `idempotency_key` for dedup |
| `direction` | enum | `inbound` \| `outbound` |
| `sender.id` | string | Platform-scoped user ID |
| `content_type` | enum | `text`, `image`, `video`, `audio`, `file`, `location`, `sticker`, `template`, `unsupported` |
| `content` | object | Type-specific content (same variant shapes as Proposal 3) |
| `reply_to_event_id` | ULID? | Canonical `event_id` of the parent message event |

**`metadata` fields:**

| Field | Type | Description |
|---|---|---|
| `adaptor` | string | Which adaptor emitted this |
| `adaptor_version` | semver | For diagnosing schema drift across deployments |
| `schema_version` | string | `{domain}/{version}` ŌĆö used by consumers for version routing |
| `idempotency_key` | string | `{platform}:{platform_message_id}` ŌĆö dedup key for DB upsert and queue exactly-once |

**Event type vocabulary:**

```
MessageReceived     ŌĆö a new inbound message arrived
MessageSent         ŌĆö an outbound message was dispatched
MessageDelivered    ŌĆö platform confirmed delivery
MessageRead         ŌĆö platform confirmed read receipt
MessageUpdated      ŌĆö platform reported an edit (e.g. Discord, Slack)
MessageDeleted      ŌĆö platform reported a deletion
MessageFailed       ŌĆö outbound send failed
```

**Pros:**
- `stream.id` as a partition key gives correct per-conversation ordering on SQS FIFO / Kafka out of the box
- `occurred_at` vs `recorded_at` enables latency SLO tracking per adaptor
- `causation_id` / `correlation_id` enable full distributed tracing across a request-reply flow
- `event_version` makes schema evolution explicit and safe ŌĆö consumers can handle v1 and v2 simultaneously
- Full event vocabulary (`Delivered`, `Read`, `Updated`, `Deleted`) future-proofs the system beyond just inbound text
- Immutability is enforced by design ŌĆö corrections are new events, audit log is free

**Cons:**
- `stream.sequence` requires a broker that supports ordered partitions (SQS FIFO, Kafka) ŌĆö plain SQS doesn't provide it
- `causation_id` / `correlation_id` are only useful if the whole system adopts them; half-adoption creates confusion
- Event vocabulary expansion (`MessageDelivered`, `MessageRead`) requires adaptor support ŌĆö most platforms don't expose these uniformly
- Heavier than necessary if the only consumer is a simple DB writer

## Comparison Summary

| | P1 CloudEvents | P2 Typed Variants | P3 Event Sourcing |
|---|---|---|---|
| **Queue fit** | Excellent ŌĆö spec-compliant | Excellent ŌĆö minimal | Excellent ŌĆö partition key built-in |
| **DB fit** | Fair ŌĆö `data` is opaque blob | Good ŌĆö content is typed | Good ŌĆö append-only log maps to timeseries/event store |
| **Type safety** | Low | High | Medium |
| **Extensibility** | High | Medium | High ŌĆö event versioning |
| **Complexity** | Low | Medium | High |
| **Dedup story** | Via `id` | Via `idempotency_key` | Via `idempotency_key` + `stream.sequence` |
| **Ordering guarantee** | None | None | Via `stream.id` partition key |
| **Outbound reuse** | Poor ŌĆö needs separate schema | Good ŌĆö `direction` field | Excellent ŌĆö full event vocabulary |
| **Beyond text messages** | Limited | Partial | Excellent ŌĆö `MessageRead`, `MessageDeleted`, etc. |
| **Best fit for** | Interop / open ecosystems | Codegen / type-safe services | High-throughput / audit / replay |
