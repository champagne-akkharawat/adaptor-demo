# Unified Message Schema — Proposals

Five schema proposals based on best practices in event-driven architecture, API design, and enterprise messaging systems. Platform-specific details deliberately excluded at this stage.

---

## Proposal 1: CloudEvents Envelope Pattern

**Inspired by**: [CloudEvents spec](https://cloudevents.io/), AWS EventBridge schema registry.

**Philosophy**: Strict separation between *routing/meta* and *content*. The envelope is platform-agnostic by definition; only `data` varies. Optimised for event queues (SQS, SNS, EventBridge).

```json
{
  "specversion": "1.0",
  "id": "01HZ9KXYZ...",
  "source": "adaptor/line",
  "type": "message.inbound.text",
  "time": "2026-03-16T10:00:00Z",
  "datacontenttype": "application/json",

  "data": {
    "sender": {
      "id": "u_abc123",
      "display_name": "Jane Doe",
      "avatar_url": "https://..."
    },
    "conversation": {
      "id": "c_xyz789",
      "type": "direct"
    },
    "content": {
      "type": "text",
      "text": "Hello world"
    },
    "attachments": [],
    "reply_to": null
  }
}
```

**Envelope fields:**

| Field | Type | Description |
|---|---|---|
| `specversion` | string | Always `"1.0"` — CloudEvents version |
| `id` | ULID/UUID | Globally unique event ID, generated by adaptor |
| `source` | string | `adaptor/{platform}` — which adaptor emitted this |
| `type` | string | Dot-separated event type: `message.inbound.{content_type}` |
| `time` | RFC3339 | When the event was emitted by the adaptor |
| `datacontenttype` | string | Always `application/json` for this system |

**`data` fields:**

| Field | Type | Description |
|---|---|---|
| `sender.id` | string | Platform-scoped user ID |
| `sender.display_name` | string? | Human-readable name if available |
| `sender.avatar_url` | string? | Profile picture URL if available |
| `conversation.id` | string | Channel/thread/DM identifier |
| `conversation.type` | enum | `direct`, `group`, `channel` |
| `content.type` | enum | `text`, `image`, `video`, `audio`, `file`, `location`, `sticker`, `unsupported` |
| `content.text` | string? | Plain text body (present when type is `text`) |
| `content.url` | string? | Media URL (present for media types) |
| `content.mime_type` | string? | MIME type for media |
| `content.caption` | string? | Caption for media |
| `attachments` | array | Secondary content items using same shape as `content` |
| `reply_to` | string? | `id` of the message being replied to |

**Pros:**
- Interoperable — any CloudEvents-compliant consumer can read the envelope
- Clean contract between adaptors and consumers
- `type` field enables consumer-side filtering at queue level without deserialising `data`

**Cons:**
- `data` is still ad-hoc JSON — no schema enforcement without a separate registry
- `specversion` is noise for internal-only systems

## Proposal 2: Canonical Domain Model (Rich Object Graph)

**Inspired by**: ActivityPub (W3C), XMPP message stanzas, enterprise canonical data model (CDM) pattern.

**Philosophy**: Model the *domain* richly. Every entity (User, Conversation, Message, Content) is a first-class object with its own identity. Favours completeness and queryability over minimalism. Suited for direct DB writes.

```json
{
  "message": {
    "id": "msg_01HZ9K...",
    "platform": "line",
    "platform_message_id": "abc123",
    "created_at": "2026-03-16T10:00:00Z",
    "received_at": "2026-03-16T10:00:01Z",
    "direction": "inbound",

    "sender": {
      "id": "usr_01HZ...",
      "platform_user_id": "Uabc123",
      "display_name": "Jane Doe",
      "avatar_url": "https://...",
      "account_type": "user"
    },

    "conversation": {
      "id": "conv_01HZ...",
      "platform_conversation_id": "Rxyz789",
      "type": "direct",
      "participants": []
    },

    "body": {
      "type": "text",
      "text": "Hello world",
      "language": "en",
      "media": null,
      "location": null,
      "template": null
    },

    "attachments": [],

    "thread": {
      "reply_to_id": null,
      "thread_id": null
    },

    "metadata": {
      "raw_platform_event": null,
      "adaptor_version": "1.0.0",
      "tags": []
    }
  }
}
```

**Key fields:**

| Field | Type | Description |
|---|---|---|
| `message.id` | ULID | Internal canonical ID |
| `message.platform` | enum | `line`, `slack`, `discord`, `whatsapp`, etc. |
| `message.platform_message_id` | string | Original ID from the platform (for dedup/idempotency) |
| `message.created_at` | RFC3339 | When the platform says the message was sent |
| `message.received_at` | RFC3339 | When the adaptor processed it |
| `message.direction` | enum | `inbound` \| `outbound` |
| `sender.account_type` | enum | `user`, `bot`, `system` |
| `body.type` | enum | `text`, `image`, `video`, `audio`, `file`, `location`, `sticker`, `template`, `unsupported` |
| `body.language` | BCP47? | Detected or declared language |
| `body.media` | object? | `{ url, mime_type, size_bytes, duration_secs, thumbnail_url }` |
| `body.location` | object? | `{ lat, lng, label }` |
| `body.template` | object? | `{ template_id, variables }` — for structured/rich messages |
| `metadata.raw_platform_event` | object? | Optional: store the original raw payload for debugging |

**Pros:**
- Both `id` (internal) and `platform_message_id` (external) are explicit — clean dedup story
- `created_at` vs `received_at` distinction is critical for ordering and latency monitoring
- `direction` field makes the same schema usable for outbound
- `metadata.raw_platform_event` allows lossless storage

**Cons:**
- Heavier object — more fields to populate/null out per adaptor
- Nested structure is more verbose to serialise onto a queue message

## Proposal 3: Typed Content Variant (Discriminated Union / Sum Type)

**Inspired by**: Protocol Buffers `oneof`, GraphQL union types, Rust enum patterns.

**Philosophy**: The message content is a *discriminated union* — exactly one content variant is present and it is fully typed. Avoids nullable fields; each content type has its own dedicated shape. Suited for strongly-typed codegen (Go structs, TypeScript discriminated unions, Protobuf).

```json
{
  "id": "01HZ9KXYZ...",
  "platform": "line",
  "platform_message_id": "abc123",
  "direction": "inbound",
  "timestamp": "2026-03-16T10:00:00Z",

  "sender": {
    "id": "Uabc123",
    "name": "Jane Doe"
  },

  "conversation": {
    "id": "Rxyz789",
    "type": "direct"
  },

  "content": {
    "kind": "text",
    "text": {
      "body": "Hello world",
      "preview": null
    }
  },

  "reply_to": null,
  "idempotency_key": "line:abc123"
}
```

**Content variant shapes** — exactly one `kind` is present, and only its corresponding field is populated:

```json
{ "kind": "text",     "text":     { "body": "...", "preview": null } }
{ "kind": "image",    "image":    { "url": "...", "mime_type": "image/jpeg", "width": 1280, "height": 720, "caption": null } }
{ "kind": "video",    "video":    { "url": "...", "mime_type": "video/mp4", "duration_secs": 30, "thumbnail_url": "..." } }
{ "kind": "audio",    "audio":    { "url": "...", "mime_type": "audio/ogg", "duration_secs": 12 } }
{ "kind": "file",     "file":     { "url": "...", "mime_type": "...", "filename": "report.pdf", "size_bytes": 204800 } }
{ "kind": "location", "location": { "lat": 35.6762, "lng": 139.6503, "label": "Tokyo Station" } }
{ "kind": "sticker",  "sticker":  { "sticker_id": "...", "package_id": "...", "url": null } }
{ "kind": "template", "template": { "template_id": "order_confirm", "variables": { "order_id": "ORD-99" } } }
{ "kind": "unsupported", "unsupported": { "raw_type": "custom_widget", "description": "Platform-specific content not mapped to canonical schema" } }
```

**Top-level fields:**

| Field | Type | Description |
|---|---|---|
| `id` | ULID | Canonical ID generated by adaptor |
| `platform` | enum | Source platform identifier |
| `platform_message_id` | string | Platform's own message ID |
| `direction` | enum | `inbound` \| `outbound` |
| `timestamp` | RFC3339 | Platform-reported send time |
| `sender.id` | string | Platform-scoped user ID |
| `sender.name` | string? | Display name if available |
| `conversation.id` | string | Channel/DM identifier |
| `conversation.type` | enum | `direct`, `group`, `channel` |
| `content.kind` | enum | Discriminator — determines which sibling field is present |
| `reply_to` | string? | Canonical `id` of parent message |
| `idempotency_key` | string | `{platform}:{platform_message_id}` — used for dedup at DB/queue level |

**Pros:**
- No nullable fields within a content variant — every present field is meaningful
- Trivially maps to Go `interface{}` + type switch, TS discriminated union, or Protobuf `oneof`
- `idempotency_key` is explicit and composite — prevents double-processing across restarts
- Minimal top-level footprint — fast to serialise onto SQS

**Cons:**
- Adding a new content type requires schema changes in all consumers (coordinated deploy)
- `unsupported` is a catch-all that can hide mapping gaps if not monitored

## Proposal 4: ActivityStreams 2.0 / Semantic Object Model

**Inspired by**: [W3C ActivityStreams 2.0](https://www.w3.org/TR/activitystreams-core/), ActivityPub, JSON-LD.

**Philosophy**: Messages are *semantic objects* with a formal vocabulary. An inbound message is a `Create` activity performed by an `Actor` on an `Object` (the message content) within a `Place` (the conversation). Meaning is encoded in the structure, not just field names. Suited for federation, open standards compliance, or systems that need to interop with Mastodon/ActivityPub ecosystems.

```json
{
  "@context": "https://www.w3.org/ns/activitystreams",
  "id": "https://internal/events/01HZ9K...",
  "type": "Create",
  "published": "2026-03-16T10:00:00Z",

  "instrument": {
    "type": "Service",
    "name": "line-adaptor",
    "version": "1.0.0"
  },

  "actor": {
    "type": "Person",
    "id": "https://internal/users/line:Uabc123",
    "name": "Jane Doe",
    "icon": { "type": "Image", "url": "https://..." }
  },

  "target": {
    "type": "Group",
    "id": "https://internal/conversations/line:Rxyz789",
    "name": null
  },

  "object": {
    "type": "Note",
    "id": "https://internal/messages/line:msg123",
    "content": "Hello world",
    "mediaType": "text/plain",
    "inReplyTo": null,
    "attachment": []
  },

  "x-platform": {
    "name": "line",
    "message_id": "msg123",
    "direction": "inbound"
  }
}
```

**ActivityStreams type mappings for `object.type`:**

| Canonical content | AS2 type | Notes |
|---|---|---|
| Text message | `Note` | `content` holds the text |
| Image | `Image` | `url` holds the media URL |
| Video | `Video` | `url` + `duration` |
| Audio | `Audio` | `url` + `duration` |
| File | `Document` | `url` + `mediaType` + `name` |
| Location | `Place` | `latitude`, `longitude`, `name` |
| Sticker | `Emoji` | Non-standard; falls back to `Note` with `mediaType: image/webp` |
| Template/rich | `Article` | `content` is structured HTML or JSON summary |
| Unsupported | `Object` | Generic fallback |

**`x-platform` extension fields** (prefixed to avoid namespace collision):

| Field | Type | Description |
|---|---|---|
| `x-platform.name` | enum | Source platform |
| `x-platform.message_id` | string | Platform's original message ID (for dedup) |
| `x-platform.direction` | enum | `inbound` \| `outbound` |

**Pros:**
- Fully standardised vocabulary — zero ambiguity about what `actor`, `object`, `target` mean
- `id` is a URI, which gives globally unique, resolvable identifiers for free
- Natively extensible via JSON-LD `@context` without breaking existing consumers
- Future-proof: directly compatible with ActivityPub federation if needed

**Cons:**
- Verbose — `@context`, URI-based IDs, and nested objects add significant overhead
- AS2 vocabulary doesn't map cleanly to all message types (stickers, templates need workarounds)
- URI-based IDs are awkward for internal SQS/DB use without a resolver layer
- Steep learning curve; most backend engineers are not familiar with AS2

## Proposal 5: Event Sourcing / Append-only Log Model

**Inspired by**: CQRS/Event Sourcing (Greg Young), Kafka message conventions, EventStoreDB schema patterns.

**Philosophy**: A message is an *immutable domain event* that records a fact — "a message was received". It carries full causation and correlation context, a stream partition key, and a sequence position. The schema is designed for an append-only log (Kafka, SQS FIFO, Kinesis) where order, dedup, and replay are first-class concerns. The canonical message is never mutated; corrections are new events.

```json
{
  "event_id": "01HZ9KXYZ...",
  "event_type": "MessageReceived",
  "event_version": 1,
  "occurred_at": "2026-03-16T10:00:00Z",
  "recorded_at": "2026-03-16T10:00:01Z",

  "stream": {
    "id": "conversation:line:Rxyz789",
    "sequence": 142
  },

  "causation_id": null,
  "correlation_id": "sess_01HZ...",

  "payload": {
    "platform": "line",
    "platform_message_id": "msg123",
    "direction": "inbound",

    "sender": {
      "id": "Uabc123",
      "name": "Jane Doe"
    },

    "content_type": "text",
    "content": {
      "text": "Hello world"
    },

    "reply_to_event_id": null
  },

  "metadata": {
    "adaptor": "line-adaptor",
    "adaptor_version": "1.0.0",
    "schema_version": "messages/v1",
    "idempotency_key": "line:msg123"
  }
}
```

**Top-level event fields:**

| Field | Type | Description |
|---|---|---|
| `event_id` | ULID | Globally unique event ID — monotonic, sortable |
| `event_type` | string | PascalCase domain event name: `MessageReceived`, `MessageUpdated`, `MessageDeleted` |
| `event_version` | int | Schema version of this event type — enables consumer-side migration |
| `occurred_at` | RFC3339 | When the event *happened* (platform time) |
| `recorded_at` | RFC3339 | When the adaptor *recorded* it — delta = processing latency |
| `stream.id` | string | Partition key: `{entity_type}:{platform}:{id}` — routes to the same consumer partition |
| `stream.sequence` | int? | Position within the stream — set by the queue/broker, null if not yet assigned |
| `causation_id` | ULID? | `event_id` of the event that *caused* this one (e.g. a command that triggered a reply) |
| `correlation_id` | string? | Groups related events across a user session or workflow |

**`payload` fields:**

| Field | Type | Description |
|---|---|---|
| `platform` | enum | Source platform |
| `platform_message_id` | string | Platform's ID — used with `idempotency_key` for dedup |
| `direction` | enum | `inbound` \| `outbound` |
| `sender.id` | string | Platform-scoped user ID |
| `content_type` | enum | `text`, `image`, `video`, `audio`, `file`, `location`, `sticker`, `template`, `unsupported` |
| `content` | object | Type-specific content (same variant shapes as Proposal 3) |
| `reply_to_event_id` | ULID? | Canonical `event_id` of the parent message event |

**`metadata` fields:**

| Field | Type | Description |
|---|---|---|
| `adaptor` | string | Which adaptor emitted this |
| `adaptor_version` | semver | For diagnosing schema drift across deployments |
| `schema_version` | string | `{domain}/{version}` — used by consumers for version routing |
| `idempotency_key` | string | `{platform}:{platform_message_id}` — dedup key for DB upsert and queue exactly-once |

**Event type vocabulary:**

```
MessageReceived     — a new inbound message arrived
MessageSent         — an outbound message was dispatched
MessageDelivered    — platform confirmed delivery
MessageRead         — platform confirmed read receipt
MessageUpdated      — platform reported an edit (e.g. Discord, Slack)
MessageDeleted      — platform reported a deletion
MessageFailed       — outbound send failed
```

**Pros:**
- `stream.id` as a partition key gives correct per-conversation ordering on SQS FIFO / Kafka out of the box
- `occurred_at` vs `recorded_at` enables latency SLO tracking per adaptor
- `causation_id` / `correlation_id` enable full distributed tracing across a request-reply flow
- `event_version` makes schema evolution explicit and safe — consumers can handle v1 and v2 simultaneously
- Full event vocabulary (`Delivered`, `Read`, `Updated`, `Deleted`) future-proofs the system beyond just inbound text
- Immutability is enforced by design — corrections are new events, audit log is free

**Cons:**
- `stream.sequence` requires a broker that supports ordered partitions (SQS FIFO, Kafka) — plain SQS doesn't provide it
- `causation_id` / `correlation_id` are only useful if the whole system adopts them; half-adoption creates confusion
- Event vocabulary expansion (`MessageDelivered`, `MessageRead`) requires adaptor support — most platforms don't expose these uniformly
- Heavier than necessary if the only consumer is a simple DB writer

## Comparison Summary

| | P1 CloudEvents | P2 Domain Model | P3 Typed Variants | P4 ActivityStreams | P5 Event Sourcing |
|---|---|---|---|---|---|
| **Queue fit** | Excellent — spec-compliant | Good — slightly heavy | Excellent — minimal | Poor — too verbose | Excellent — partition key built-in |
| **DB fit** | Fair — `data` is opaque blob | Excellent — maps to relations | Good — content is typed | Poor — URI IDs awkward | Good — append-only log maps to timeseries/event store |
| **Type safety** | Low | Medium | High | Low | Medium |
| **Extensibility** | High | High | Medium | Very high (JSON-LD) | High — event versioning |
| **Complexity** | Low | High | Medium | Very high | High |
| **Dedup story** | Via `id` | Via `platform_message_id` | Via `idempotency_key` | Via URI `id` | Via `idempotency_key` + `stream.sequence` |
| **Ordering guarantee** | None | None | None | None | Via `stream.id` partition key |
| **Outbound reuse** | Poor — needs separate schema | Good — `direction` field | Good — `direction` field | Good — `direction` in extension | Excellent — full event vocabulary |
| **Beyond text messages** | Limited | Partial | Partial | Good | Excellent — `MessageRead`, `MessageDeleted`, etc. |
| **Best fit for** | Interop / open ecosystems | Rich querying / analytics | Codegen / type-safe services | Federation / open standards | High-throughput / audit / replay |
