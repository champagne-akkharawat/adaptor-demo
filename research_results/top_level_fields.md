# Top-Level Fields — Canonical Message Schema

Findings from analysis of all 8 target platforms (Discord, Facebook Messenger, Instagram, LINE, Slack, Microsoft Teams, Twitter/X, WhatsApp) and best practices in event-driven CS hub architecture.

---

## Core Envelope

These fields are present on every event regardless of platform, direction, or content type.

```json
"id":                  "01HZAB...",
"event_type":          "MessageReceived",
"platform":            "slack",
"platform_message_id": "slack_ts_1710582000.123456",
"idempotency_key":     "slack:slack_ts_1710582000.123456",
"direction":           "inbound",
"occurred_at":         "2026-03-16T10:00:00Z",
"received_at":         "2026-03-16T10:00:00.201Z",
"correlation_id":      "sess_01HZAA...",
"adaptor_version":     "1.2.0"
```

### `id`
**Type**: ULID string

**Why**: Every downstream system needs a stable handle independent of the platform. Platform IDs are only unique *within* a platform — you can't use a Slack timestamp as a primary key alongside a LINE message ID without collision risk.

**When/How**: Generated once by the adaptor immediately after signature verification, before any further mapping. Never regenerated — if the same webhook arrives twice, the second pass hits the idempotency check and is dropped before a new `id` is assigned.

---

### `event_type`
**Type**: Enum string

**Why**: Enables consumers to filter and route at the queue level without deserialising the payload body. A consumer subscribed to the unified queue can drop `MessageDeleted` events via SQS filter policy or Kafka routing before they reach application code. Also makes the schema cover the full event lifecycle beyond inbound text.

**Vocabulary**:
```
MessageReceived    — new inbound message
MessageUpdated     — user edited a message (Slack, Discord)
MessageDeleted     — user deleted a message (Slack, Discord, Teams)
MessageDelivered   — platform confirmed delivery (WhatsApp, FB Messenger)
MessageRead        — platform confirmed read receipt (WhatsApp, Instagram)
MessageFailed      — outbound send failed
```

**When/How**: Set statically per code path — webhook handler always produces `MessageReceived`; delete event handler always produces `MessageDeleted`; etc.

---

### `platform`
**Type**: Enum string — `discord | slack | teams | line | facebook | instagram | twitter | whatsapp`

**Why**: By the time a message reaches the hub core or a queue consumer, the source is no longer implied by the network path. Without this field, a consumer cannot know which reply API to call, which identity resolution table to query, or which policy rules apply (24h window check is only relevant for `facebook`, `instagram`, `whatsapp`).

**When/How**: Hardcoded constant in each adaptor. Never varies at runtime.

---

### `platform_message_id`
**Type**: String

**Why**: Three uses — (1) constructing `idempotency_key`, (2) linking back to the source platform for audit and debugging, (3) threading on platforms that reference messages by their native ID (Discord, Slack).

**When/How**: Extracted from the inbound webhook. Never generated by the adaptor.

| Platform | Source field |
|---|---|
| Slack | `event.ts` |
| Discord | `d.id` |
| Teams | `activity.id` |
| LINE | `events[].message.id` |
| Facebook / Instagram | `entry[].messaging[].message.mid` |
| WhatsApp | `messages[].id` (`wamid.*`) |
| Twitter/X | `direct_message_events[].id` |

---

### `idempotency_key`
**Type**: String — `"{platform}:{platform_message_id}"`

**Why**: All platforms guarantee at-least-once delivery. The same webhook will arrive 2–3 times during network blips or retries. Used as a `UNIQUE` DB constraint, SQS FIFO `MessageDeduplicationId`, or Redis SET member. Composite (platform + id) because message IDs are only unique within a platform.

**When/How**: Constructed by the adaptor immediately after `platform_message_id` is extracted.

```
idempotency_key = platform + ":" + platform_message_id
// "slack:1710582000.123456"
// "whatsapp:wamid.HBgNNjE0..."
```

---

### `direction`
**Type**: Enum — `inbound | outbound`

**Why**: The same schema covers both messages received from a user and messages sent to a user. A CS conversation thread contains both. Without this field, a consumer rendering a conversation timeline cannot distinguish which side a message belongs to. Also gates business logic — policy window checks and template enforcement only apply in specific directions.

**When/How**: Hardcoded per code path. Webhook handlers → `inbound`. Send handlers → `outbound`.

---

### `occurred_at`
**Type**: RFC3339 timestamp

**Why**: The user-perceived send time. Used for: (1) ordering messages correctly within a conversation during catch-up after downtime, (2) computing `policy.window_expires_at` (`occurred_at + 24h`), (3) displaying timestamps in the agent UI. Must come from the platform — the adaptor may process events out of order or with delay.

**When/How**: Extracted from the platform payload. Each platform exposes it differently:

| Platform | Field | Format |
|---|---|---|
| Slack | `event.ts` | Unix float string → ×1000 |
| Discord | `d.timestamp` | ISO8601 |
| Teams | `activity.timestamp` | ISO8601 |
| LINE | `events[].timestamp` | Unix ms integer |
| Facebook / Instagram | `entry[].messaging[].timestamp` | Unix ms integer |
| WhatsApp | `messages[].timestamp` | Unix seconds integer |
| Twitter/X | `created_timestamp` | Unix ms string |

---

### `received_at`
**Type**: RFC3339 timestamp

**Why**: `received_at − occurred_at` = end-to-end delivery latency. This is the primary SLO metric per platform. Distinguishes platform-level delays (WhatsApp batch-delivering queued messages) from adaptor processing bottlenecks.

**When/How**: Set by the adaptor immediately before publishing to the queue — the last field written, capturing full adaptor processing time.

```
received_at = time.Now().UTC()
```

---

### `correlation_id`
**Type**: ULID string (nullable)

**Why**: Groups all events belonging to the same customer interaction — initial message, bot reply, agent takeover, resolution. In a CS hub this maps to a ticket or session. Without it, reconstructing a full interaction chain requires expensive joins on `conversation.id` + time range.

**When/How**: Set by the hub core on the first `MessageReceived` for a conversation session; carried forward on all subsequent events in that session. Null on the very first event until the hub assigns it.

---

### `adaptor_version`
**Type**: Semver string

**Why**: When a bad adaptor deploy introduces a mapping bug (wrong timestamp parsing, mangled IDs), you need to identify affected messages quickly. Enables precise incident queries:

```sql
SELECT * FROM messages
WHERE platform = 'whatsapp'
  AND adaptor_version = '1.2.0'
  AND received_at BETWEEN '...' AND '...'
```

**When/How**: Hardcoded constant in the adaptor, bumped on each release.

## Sender

```json
"sender": {
  "id":          "cust_01HZ...",
  "platform_id": "U12345",
  "name":        "Jane Doe",
  "type":        "user"
}
```

| Field | Why |
|---|---|
| `id` | Internal resolved customer identity — one record across all platforms |
| `platform_id` | Raw platform ID. Kept separate because the same person has *different* IDs per platform (PSID on Facebook, IGSID on Instagram, phone number on WhatsApp) |
| `name` | Display name for the CS agent UI |
| `type` | `user / bot / system` — filters out bot echoes; Slack echoes your own app's messages back |

---

## Conversation

```json
"conversation": {
  "id":           "conv_01HZ...",
  "platform_id":  "C08AB1234",
  "type":         "channel",
  "thread_id":    null,
  "workspace_id": "T01ABCDEF"
}
```

| Field | Why |
|---|---|
| `id` | Canonical conversation ID — partition key for message ordering |
| `platform_id` | Raw channel/DM/room ID from the platform |
| `type` | `direct / group / channel` — LINE sends all three types through the same webhook |
| `thread_id` | Slack and Teams have threads as distinct addressable objects inside a channel; a reply inside a thread uses the thread ID, not the parent message ID |
| `workspace_id` | Slack (workspace), Teams (tenant), Discord (guild) — platforms with a multi-tenant hierarchy above the conversation |

## Content

Discriminated union — exactly one variant is present, fully typed, no nullable fields within a variant.

```json
"content": {
  "kind": "text | image | video | audio | file | location | sticker | template | unsupported",
  "<kind>": { ... }
}
```

See `proposals/unified_schema.md` Proposal 2 for full variant shapes.

## Threading

```json
"reply_to":  "01HZAA...",
"thread_id": null
```

| Field | Why |
|---|---|
| `reply_to` | Canonical ID of the parent message being replied to |
| `thread_id` | Slack/Teams: threads are addressable objects; replies inside a thread reference `thread_id`, not `reply_to` — both concepts must be represented |

## Policy

```json
"policy": {
  "window_expires_at": "2026-03-17T10:00:00Z",
  "can_send_freeform":  true,
  "requires_template":  false
}
```

| Field | Driven by |
|---|---|
| `window_expires_at` | Facebook Messenger, Instagram, WhatsApp — 24h reply window. The CS agent UI needs to show a countdown; outbound handler must check before sending |
| `can_send_freeform` | WhatsApp outside the 24h window — only pre-approved templates allowed |
| `requires_template` | Same constraint surfaced as a UI prompt: "use a template" |

**When/How**: Computed by the adaptor from `occurred_at` at ingest time. Null for platforms with no messaging window (Slack, Teams, Discord).

## Platform Context

Outbound routing data captured from the inbound event. Must be stored on the message because it cannot be reconstructed later.

```json
"platform_context": {
  "reply_token":    "nHuyWiB7yP5Zw52FIkcQobQuGDXCTA",
  "service_url":    "https://smba.trafficmanager.net/...",
  "activity_id":    "f:abc123",
  "messaging_type": "RESPONSE",
  "page_id":        "123456789"
}
```

| Field | Driven by |
|---|---|
| `reply_token` | LINE — one-time token that expires; must be captured immediately from the inbound event |
| `service_url` | Teams — reply endpoint is dynamic per tenant; cannot be hardcoded |
| `activity_id` | Teams — required to post a reply to a specific activity |
| `messaging_type` | Facebook Messenger — `RESPONSE / UPDATE / MESSAGE_TAG` must be declared on outbound; value depends on why you are replying |
| `page_id` | Facebook Messenger, Instagram — outbound API calls require the page ID, not just the sender ID |

## Routing

Written by the hub's routing rules engine after ingestion, not by the adaptor. Included in the schema so the message document is the single source of truth for the CS agent UI.

```json
"routing": {
  "ticket_id":   "ticket_01HZ...",
  "assigned_to": "agent_01HZ...",
  "queue":       "support-tier1",
  "tags":        ["billing", "urgent"]
}
```

| Field | Why |
|---|---|
| `ticket_id` | Links the message to a CS ticket |
| `assigned_to` | Which agent or team owns this conversation |
| `queue` | Which routing queue the message was assigned to |
| `tags` | Applied by routing rules for filtering, SLA classification, and reporting |

---

## Fields Considered and Excluded

| Field | Reason excluded |
|---|---|
| `schema_version` | Redundant given Tolerant Reader pattern + Expand/Contract schema evolution strategy |
| `tenant_id` | Only relevant for multi-tenant SaaS deployments; add at that point, not speculatively |
| `causation_id` | Only useful if the whole system adopts it; half-adoption creates noise without value |
| `retry_count` | Belongs on queue message metadata, not the canonical event payload |
| `raw_payload` | Too large for the envelope; store separately keyed by `id` if needed for debugging |
